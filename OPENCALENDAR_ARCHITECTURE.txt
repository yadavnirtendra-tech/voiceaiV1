====================================================================
OpenCalendar - Unified Calendar Sync Engine
Complete Architecture, Setup & Deployment Guide
====================================================================

1. SYNC DURATION & TIMING
--------------------------------------------------------------------
OpenCalendar is a "Real-Time" synchronization engine. It does NOT rely on slow interval-based polling (like syncing every 15 minutes). Instead, it uses an event-driven webhook architecture:

* Real-Time Webhooks: 
  When an event is created, updated, or deleted on Google or Outlook, their respective servers instantly send an HTTP POST payload to our backend webhook routes. The sync is usually processed and reflected within 3-10 seconds.
* Hourly Maintenance (Cron): 
  Webhooks naturally expire (e.g., Microsoft subscriptions expire every 3 days, Google every 7 days). The backend runs a Node-Cron job every 1 hour (0 * * * *) to proactively renew any webhooks that are nearing expiration.
* Initial & Manual Sync: 
  A full deep-sync covering a 30-day window runs automatically the moment an account is connected via OAuth, or when the user clicks "Sync Now" on the dashboard.

2. ARCHITECTURE & PROCESS FLOWCHART
--------------------------------------------------------------------
OpenCalendar operates on a strict State Machine to prevent infinite sync loops across different providers.

[External Calendar (Google/Microsoft)]
         |
         | (1) Real-Time Webhook HTTP POST
         v
[Webhook Listener Route] 
         |
         | (2) Queues Event Processing
         v
[State 1: FETCH EVENT] 
         | --> Fetches the Delta (latest changes) using Sync Tokens.
         v
[State 2: LOOP GUARD] 
         | --> Checks "Extended Properties" metadata. 
         | --> If tagged as "CalendarSync Block", it drops the event to prevent an infinite bouncing loop.
         v
[State 3: DATABASE UPSERT] 
         | --> Saves the canonical event to PostgreSQL.
         v
[State 4: CONFLICT RESOLUTION] 
         | --> Checks if the new event double-books an existing slot.
         v
[State 5: SHADOW BLOCKING] 
         | --> Connects to the *opposite* provider.
         | --> Translates the native timezone mathematically.
         | --> Injects a "Reserved" Shadow Block event.
         v
[State 6: SYNC LOGGING] 
         | --> Records success/failure into the Prisma database for the Dashboard feed.


3. COMPLETE APP DESCRIPTION
--------------------------------------------------------------------
OpenCalendar is a multi-tenant SaaS backend built with Express.js, Prisma, and PostgreSQL. Its primary goal is to maintain a "Single Source of Truth" for user availability across fragmented calendar ecosystems (Google Workspace and Microsoft 365). 

Key Features:
* Zero-Friction OAuth: Connects to Google and Azure AD without users needing their own developer credentials.
* Bidirectional Sync: Changes in Microsoft reflect in Google; changes in Google reflect in Microsoft.
* Native Timezone Engine: Dynamically maps the origin timezone across APIs so events never shift hours out of place.
* Dark Mode Premium UI: Pure vanilla HTML/CSS dashboard for extreme speed and security.


4. LOCAL SETUP & INSTALLATION
--------------------------------------------------------------------
Prerequisites: Node.js (v18+), PostgreSQL Database.

Step 1: Clone the Repository
  git clone https://github.com/yadavnirtendra-tech/autocalender.git
  cd autocalender/backend

Step 2: Install Dependencies
  npm install

Step 3: Environment Variables
  Create a `.env` file in the backend directory:
  DATABASE_URL="postgresql://user:password@localhost:5432/opencalendar"
  GOOGLE_CLIENT_ID="your_google_client_id"
  GOOGLE_CLIENT_SECRET="your_google_secret"
  MICROSOFT_CLIENT_ID="your_ms_client_id"
  MICROSOFT_CLIENT_SECRET="your_ms_secret"
  JWT_SECRET="secure_random_string"
  NODE_ENV="development"
  FRONTEND_URL="http://localhost:5173"
  API_BASE_URL="http://localhost:3000"

Step 4: Initialize Database
  npx prisma generate
  npx prisma db push

Step 5: Start the Application
  npm run dev


5. DEPLOYMENT PROCESS (Production)
--------------------------------------------------------------------
OpenCalendar is designed to run in a stateless containerized environment (like Railway, Render, or Heroku) connected to a managed PostgreSQL database.

Railway Deployment (Recommended):
1. Create a Railway.app account.
2. Provision a new "PostgreSQL" database instance in Railway.
3. Create a new "Empty Service" and link your GitHub repository.
4. Set the Root Directory to `/backend`.
5. Under Variables, map the DATABASE_URL to your new PostgreSQL instance.
6. Add the production OAuth keys and set NODE_ENV="production".
7. Set the start command:
   npm install && npx prisma generate && npx prisma db push && npm start

Vercel Deployment (Frontend):
Since the frontend is vanilla HTML/CSS/JS located in `frontend/public`, you can deploy it instantly:
1. Connect Vercel to your GitHub repo.
2. Set the Root Directory to `frontend`.
3. Vercel will automatically serve the `public` directory as a static site.

Critical Deployment Commands:
* Apply Database Migrations: `npx prisma migrate deploy`
* Start Server: `node src/index.js`
* Monitor Logs: `tail -f console.log` (or check Railway dashboard logs)

6. SECURITY CONSIDERATIONS
--------------------------------------------------------------------
* OAuth tokens (Access & Refresh tokens) are encrypted at rest in the database using AES-256-GCM.
* Webhook payloads are verified against HMAC signatures and specific Azure/Google validation tokens.
* API rate limiting is enforced via `express-rate-limit` to prevent brute-force attacks on the sync engine.


7. SCALABILITY, USAGE & COST ESTIMATION (SaaS MODEL)
--------------------------------------------------------------------
Understanding the load is critical for scaling OpenCalendar as a SaaS application. The app's architecture operates efficiently, but high-volume users will generate predictable API and database load.

[A] Usage Profile Analysis (Extreme Edge Case)
Consider a highly active "Power User" who has 300 meetings per day on Google Calendar and 300 meetings per day on Microsoft Outlook (Total: 600 meeting creations/updates per day).

For *each* individual meeting update, the system executes:
1. Inbound Webhook (1 HTTP Request)
2. Provider API Fetch (1 External GET Request)
3. Shadow Block Creation (1 External POST Request to the opposite provider)
4. Database Operations (~5 Queries: Read duplicate check, Upsert Source Event, Conflict Check, Insert ShadowBlock, Insert SyncLog)

Daily Load per Power User (600 updates):
* Inbound Webhooks: 600 requests/day
* External API Calls: 1,200 calls/day (600 Google, 600 Microsoft)
* Database Operations: ~3,000 queries/day

[B] Scaling to 1,000 Power Users (SaaS Projections)
If you acquire 1,000 Power Users, each generating 600 updates daily:
* Total Webhook Traffic: 600,000 incoming requests/day (~7 requests per second).
* External API Traffic: 1,200,000 requests/day.
* Database Traffic: 3,000,000 queries/day (~35 queries per second).

[C] API Limitations & Quotas
* Google Calendar API: The free tier allows 1,000,000 queries per day. At 1,000 power users, you will hit this quota and need to request an expansion from Google Cloud Console (which is generally free upon verification).
* Microsoft Graph API: Uses dynamic throttling rather than strict hard caps, but easily handles this load across 1,000 unique tenants.

[D] Infrastructure Cost Estimates (Monthly)
Because Node.js handles asynchronous webhook IO exceptionally well, backend compute costs are incredibly low. The primary cost will be the database storage and query capacity.

1. Compute (e.g., Railway/Render Container):
   - A standard $10 to $20/month container (1GB RAM, 1 vCPU) can easily handle 7 incoming webhook requests per second.
   - Cost: ~$15/month

2. Database (e.g., Managed PostgreSQL on Railway/Supabase):
   - 35 queries per second requires decent IOPS. Because event metadata and logs accumulate, storage will grow over time. You will need a standard tier.
   - Cost: ~$25 to $50/month

3. Total Operating Cost:
   - For 1,000 extreme Power Users, the raw infrastructure cost will be approximately $40 to $70/month. If charging $5/user/month for the SaaS, the profit margins are exceptionally high (>98%).
